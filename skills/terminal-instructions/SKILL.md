Триггеры - выдача любых shell-команд, инструкции по установке через CLI, команды для git push/pull/commit, команды для запуска скриптов, инструкции по работе с файлами через терминал. Скил заставляет тебя явно помечать какие команды выполнять построчно а какие можно одним блоком, и предупреждать об интерактивных запросах (пароли, Y/N, выбор опций).

Ты выдаёшь команды для терминала чётко и понятно для человека который не разработчик. Главное правило - не давай ученику свалиться на середине последовательности команд потому что одна из них требует ввода или ждёт ответа другой.

ПРАВИЛО ПОСТРОЧНО ИЛИ БЛОКОМ

ВСЕГДА явно говори как выполнять команды. Не оставляй на догадку.

Команды можно одним блоком если -
- Они НЕ зависят друг от друга (каждая выполняется самостоятельно)
- НЕ требуют интерактивного ввода между ними (пароля, Y/N, выбора)
- Не нужно проверять результат одной перед запуском следующей
- Это обычная последовательность `cd → git add → git commit → git push` без сюрпризов

Команды НАДО давать построчно если -
- Какая-то из них может попросить пароль, токен, Y/N
- Результат одной нужно скопировать в следующую (например URL репо после `gh repo create`)
- Одна команда может зависнуть или потребовать ручного нажатия
- Если что-то пойдёт не так - нужно остановиться и разобраться, а не лететь дальше
- Установка чего-то нового (brew install, npm install -g) где могут вылезти ошибки прав доступа

КАК ПОМЕЧАТЬ В ТЕКСТЕ

Когда выдаёшь блок команд который можно одним пуском - пишешь сверху -

«Запусти всё одним блоком, скопируй разом -»

Когда выдаёшь команды построчно - пишешь сверху -

«Выполняй построчно. Каждую команду запускай отдельно и смотри что отвечает терминал перед следующей -»

Если в блоке есть интерактивные команды - предупреди что произойдёт -

«Команда `git push` может попросить логин и пароль. Username - твой логин GitHub. Password - это НЕ пароль от GitHub, а Personal Access Token. Если терминал ничего не спросит - значит токен уже сохранён в Keychain, проходим дальше.»

«Команда `brew install gh` спросит подтверждение [Y/n]. Нажми Enter или Y.»

«Команда `sudo` попросит пароль от Mac. Это пароль которым ты разлочиваешь компьютер.»

ПРИМЕРЫ ПРАВИЛЬНОЙ ВЫДАЧИ

Пример 1. Простой git push - можно блоком.

«Запусти всё одним блоком -

```
cd ~/Documents/Claude/Projects/проект
git add .
git commit -m "сообщение"
git push
```

Команда `git push` может попросить логин/токен один раз. После этого работает молча.»

Пример 2. Установка GitHub CLI - построчно.

«Выполняй построчно. После каждой команды смотри что отвечает терминал -

`brew install gh` - установка займёт 30-60 секунд. Если попросит [Y/n] - нажми Enter.

`gh auth login` - запустит интерактивную авторизацию. Выберешь GitHub.com → HTTPS → Y (auth Git с твоими credentials) → Login with web browser. Откроется браузер.

`gh repo create my-repo --public --source=. --push` - после успешной авторизации создаст репо и сразу запушит.»

Пример 3. Создание нового репо через сайт + push - построчно с проверкой.

«Сначала создай репо на github.com через сайт (это руками в браузере, не в терминале). После этого вернись в терминал и выполняй построчно -

`git remote remove origin` - убирает старый (некорректный) remote если он был. Если remote ещё не настроен - будет ошибка fatal: No such remote 'origin', это нормально, идём дальше.

`git remote add origin https://github.com/ТВОЙ-ЛОГИН/имя-репо.git` - ВАЖНО. Подставь свой реальный логин GitHub, не пиши буквально ТВОЙ-ЛОГИН.

`git push -u origin main` - запустит первый пуш. Может попросить логин и токен один раз. После этого работает молча.»

КАК НАЗЫВАТЬ PLACEHOLDER В КОМАНДАХ

Если в команде есть кусок который пользователь должен заменить на своё (логин, имя репо, путь) - явно помечай его и предупреждай что НЕ копировать дословно. Иначе пользователь скопирует «ТВОЙ-ЛОГИН» буквально и команда не сработает.

Хорошо -
«В команде ниже ЗАМЕНИ `ТВОЙ-ЛОГИН` на свой реальный логин GitHub (например slavikpicurin464-cmd). Не копируй дословно слово ТВОЙ-ЛОГИН -

`git remote add origin https://github.com/ТВОЙ-ЛОГИН/repo.git`»

Плохо -
«Выполни -

`git remote add origin https://github.com/ТВОЙ-ЛОГИН/repo.git`»

ЧТО ДЕЛАТЬ ЕСЛИ ОШИБКА

Когда даёшь команды - добавляй короткую подсказку что делать если что-то пошло не так. Не обширную, а одну строку с типичной ошибкой.

«Если выдаст `command not found: gh` - значит GitHub CLI не установлен, поставь через `brew install gh` сначала.»

«Если `permission denied` - запусти команду с `sudo` в начале.»

«Если `Repository not found` - проверь что в команде твой логин GitHub а не плейсхолдер.»

ЧЕГО НЕ ДЕЛАТЬ

Не давай 5-7 команд блоком если одна из них может попросить пароль. Пользователь скопирует блок целиком, первая команда попросит пароль, остальные 4-6 команд уйдут в строку запроса пароля как часть пароля - сломается всё.

Не давай команду с `rm -rf` или `sudo rm` без явного предупреждения «эта команда удаляет данные, дважды проверь путь перед запуском».

Не предполагай что у пользователя есть установленное окружение (git / gh / brew / node / python). Если используешь команду которая требует установки - сначала проверь, либо дай команду на установку первой.

Не используй малопонятные алиасы которые могут не работать у пользователя. Используй полные команды.

КОГДА ИСПОЛЬЗОВАТЬ ДРУГИЕ СКИЛЫ

Этот скил включается автоматически при выдаче любых терминальных команд. Не нужно явно вызывать - просто помечай команды правильно когда выдаёшь.

Если задача не про терминал а про что-то другое - этот скил не подгружается.
